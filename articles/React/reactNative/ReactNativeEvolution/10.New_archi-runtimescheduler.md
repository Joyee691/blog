# RN 的事件调度 RuntimeScheduler

## 为什么需要事件调度

在新架构的综述文章中我们说过旧版本 RN 典型的渲染流程是：

```js
JS 接收 Native 的消息
  ↓
JS 执行更新逻辑
  ↓
JS 通过 Bridge 将更新指令发回 Native
  ↓
Native 触发渲染流程
```

现在我们尝试用线程的视角重新看一下这个流程：

```js
[JS thread] JS 接收 Native 的消息 -> JS 执行更新逻辑 -> JS 将更新指令发回 Native
                                                           |
                                                         Bridge
                                                           ↓
[Native thread]                                     Native 触发渲染流程(layout -> commit -> paint)
```

这个流程最大的问题就是：**JS 对后续 Native 的操作毫无感知**

这就引出了第一个问题：**需要同步获取/感知布局信息的能力（比如 ref、useLayoutEffect 等需要在 commit 后同步读取布局的能力）的执行时机无法被保证** 

接着我们来看一个例子：

```jsx
setState(A)
await Promise.resolve()
setState(B)
```

这样一段代码，在 web 的语境中，我们是不会看到中间状态 A 的

但是在 RN 中，我们有机会先看到 A，再看到 B
